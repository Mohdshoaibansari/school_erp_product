## Reasoning

C-02 is a new capability that does not exist in the current OpenSpec spec tree. The `openspec/specs/` directory contains only `tenant-institution/spec.md` (C-01). There is no `identity-user-management` spec to modify. This makes C-02's delta type ADDED exclusively — a brand-new domain.

C-02 does NOT modify C-01's behavioral spec. C-01 owns Client, Institution, OrgUnit, and InstitutionType entities with their lifecycles and the tenant isolation contract. C-02 owns User, UserProfile, UserCategory, Role, RoleAssignment, UserIdentifier, and UserLifecycleEvent entities — a completely separate set of entities with their own lifecycle and business rules. The two capabilities share kernel infrastructure (TenantContext, TenantAwareRepositoryBase, AuditEmitter) but do not modify each other's behavioral contracts.

The cross-cutting analysis requires careful judgment. C-02 references several other capabilities per the PRD §2.2 table:

- **C-01 (Tenant)** — C-02's User table has `client_id` and `institution_id`, following C-01's tenant isolation model (D1). C-02 inherits TenantAwareRepositoryBase from kernel. C-02's ownership transfer (D12) references C-02 users who migrate institutions. These are infrastructure dependencies and boundary declarations, NOT modifications to C-01's spec.
- **C-03 (Auth)** — C-02 stores users; C-03 will verify their identity and issue JWTs. Login with specific email → access specific institution (Decision 3). C-02 does not modify C-03's spec.
- **C-04 (AuthZ)** — C-02 stores role assignments; C-04 will enforce what those roles can do. C-02's RoleAssignment table is consumed by C-04's Casbin enforcement. C-02 does not modify C-04's spec.
- **C-05 (Academic Structure)** — C-02 does not model academic assignments. A teacher's subject assignment is C-05. C-02 does not modify C-05's spec.
- **C-06 (Relationships)** — C-02 does not model student-parent relationships. That is C-06. C-02 does not modify C-06's spec.
- **C-09 (Notification)** — C-02 does not send invitations or notifications. C-09 consumes C-02 user data for delivery. C-02 does not modify C-09's spec.
- **C-11 (Audit)** — C-02 emits audit events via the AuditEmitter Protocol (C-11 boundary). C-02 does not own the audit store. C-02 does not modify C-11's spec.
- **C-12 (Business Codes)** — UserIdentifier values (Student ID, Employee ID) are generated by C-12. C-02 stores them. C-02 does not modify C-12's spec.

None of these referenced capabilities have their behavioral specs modified by C-02. The references are boundary declarations within C-02's own spec — they describe C-02's external interface surface, not modifications to other domains.

C-02 OWNS its entities and their contracts outright: User, UserProfile, UserCategory, Role, RoleAssignment, UserIdentifier, UserLifecycleEvent, and the User lifecycle state machine. The references to other capabilities are recorded as boundary notes WITHIN C-02's own spec.

## ADDED requirements (high-level)

These are the requirement areas that will become requirements/scenarios in `specs/identity-user-management/spec.md` during prd-to-sdd. Each maps to PRD §5 Acceptance Criteria and grill-me decisions.

- **Tenant isolation contract for C-02 entities** — User table has `client_id` with RLS policy. Repositories inherit TenantAwareRepositoryBase. A user at School A can never see or edit user data from School B. (Decision 1, AC-1, AC-17, AC-20)
- **User CRUD + identity** — User creation (email, name, UserCategory), identity update, list/query with filters (UserCategory, Role, lifecycle status). (Decision 4, AC-4, AC-14)
- **User email uniqueness** — Each User has a UNIQUE email across the entire platform. No two User records can share the same email. Collision returns an error. (Decision 2, AC-2)
- **UserProfile (1:1, separate table)** — Extended profile fields (photo, date_of_birth, gender, blood_group) in a separate table linked 1:1 to User. Profile is optional — a User can exist without a UserProfile. (Decision 5, AC-5)
- **UserCategory lookup table** — Configurable classification (Learner, Academic Staff, Academic Leadership, Administrative Staff, Executive Leadership). Adding a new category is a data insert, not a code change. (Decision 9, AC-12, AC-14, AC-19)
- **Role lookup table** — Configurable assignment labels (Teacher, HOD, Principal, Student, Parent, Staff, Admin). Adding a new role is a data insert, not a code change. (Decision 10, AC-13, AC-15, AC-19)
- **RoleAssignment (User + Role + Scope)** — A user can hold multiple roles at the same institution. Each RoleAssignment has a `scope` field defining what the role covers (e.g., "Mathematics Department"). Institution comes from the User record — RoleAssignment does NOT store `institution_id`. (Decision 6, AC-6, AC-7)
- **UserIdentifier (institution-scoped business identifiers)** — Student ID, Employee ID, Admission Number. Typed, institution-scoped. Unique per (institution, type, value) via User.institution_id. UserIdentifier does NOT store `institution_id`. (Decision 7, AC-8, AC-9)
- **User lifecycle state machine** — `Invited → Pending → Active → Suspended → Archived`. Allowed transitions: Invited→Pending, Pending→Active, Active→Suspended, Suspended→Active, Active→Archived, Suspended→Archived. Archived is terminal (no outgoing arcs). Every transition records a lifecycle event (state, reason, actor) and is audited via C-11. (Decision 8, AC-10, AC-11)
- **UUID v4 primary keys** — All C-02 entities use UUID v4 primary keys. No autoincrement, no C-12-generated codes. (Decision 4, AC-4)
- **ModuleManifest registration** — C-02 registers via the ModuleManifest Protocol (A5). Routes registered via `register_routes`. Casbin policies registered via `register_casbin_policies` (no-op until C-04 is built). (AC-16)
- **Audit emission** — C-02 emits synchronous audit events via the AuditEmitter Protocol for: user creation, lifecycle transitions, role assignment/removal, identifier creation/deletion. (AC-18, C-11 boundary)
- **Seed data** — UserCategory and Role lookup tables have seed data in the Alembic migration (same pattern as C-01's `legal_entity_type`, `org_unit_type`, `institution_type_name` lookups). (R6)
